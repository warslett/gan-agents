## Idea: Event-Sourced Game Actors with Declarative Move Protocol

### Description

This architecture models each Rummikub game as a lightweight, event-sourced actor running in a stateful serverless environment. The design deliberately abstracts the actor model away from any single vendor: the game actor is a plain TypeScript class that receives messages, manages an event log with co-located storage, and pushes updates over WebSockets. This class can be deployed on Cloudflare Durable Objects, on AWS Lambda with DynamoDB and API Gateway WebSockets, on Rivet Actors, or even on a self-hosted Node.js process with in-memory or Redis-backed state. The key property is that each game runs as an isolated single-threaded actor with durable storage -- a pattern available on every major platform, not a vendor-specific trick. A thin platform adapter layer (under 200 lines) maps the actor interface to the host environment's storage and WebSocket APIs, keeping the entire game engine portable.

Each game actor maintains its state as an ordered log of moves (events). To avoid replay latency growing with game length, the actor persists periodic snapshots -- for example, every 10 moves -- so that state reconstruction only replays events since the last snapshot. When a player submits a move, the actor loads the most recent snapshot, replays subsequent events, validates the move, appends the event to the log, and pushes updated state to connected clients. Serverless deployments only execute when moves are made, matching the bursty nature of turn-based play where minutes may pass between actions.

The heart of the system is a declarative move protocol. During a player's turn, all tile manipulation -- dragging tiles between rack and board, splitting runs, forming new groups -- happens entirely on the client side within an isolated "sandbox" copy of the board state. When the player commits their turn, the client computes a declarative move description: a structured diff specifying which tiles moved from rack to board, which tiles were rearranged between sets, and the final composition of every set on the board. This single atomic message is sent to the server actor, which independently validates it by checking that all sets are legal, no tiles have appeared or vanished (every tile is accounted for by ID against the server's authoritative state), the initial meld requirement is met if applicable, and at least one new tile was played from the rack. If validation fails, the server rejects the move and the client reverts to the pre-turn snapshot. The server treats the client as untrusted -- the declarative diff is the trust boundary, and the server performs full independent verification of the proposed board state using authoritative tile IDs, making it impossible for a malicious client to fabricate tiles or reference tiles outside their rack. To prevent abuse, the server enforces a maximum message size on diffs (a Rummikub board has at most 106 tiles, so diffs are naturally bounded) and rate-limits move submissions per connection.

The game validation logic is written in TypeScript as a pure-function module with no runtime-specific dependencies, ensuring it runs identically on any server environment and in the browser. The client uses this shared module to provide instant feedback during tile placement (highlighting invalid sets, showing meld point totals) before the server round-trip. The server's validation is always authoritative -- client-side validation is purely a UX convenience and never trusted.

AI opponents are implemented as difficulty-tiered functions behind a simple internal API contract: given a game state (board sets and the AI player's rack), return a move in the same declarative diff format that human clients produce. A "beginner" AI uses a fast greedy heuristic that only looks for direct placements. An "expert" AI uses a more thorough search strategy (such as a backtracking solver written in pure TypeScript to avoid native dependency issues across environments). These AI functions are invoked directly by the game actor on the server side, not as separate HTTP microservices, eliminating network latency and external failure points. If an AI computation exceeds a time budget (e.g., 500ms), the game actor falls back to the greedy heuristic, ensuring turns always complete. Because the AI produces moves in the same declarative format as human players, adding new difficulty levels requires only deploying a new function variant -- the game engine and validation pipeline remain untouched.

For fault tolerance, the event log is the system's primary durability mechanism. Each game actor periodically flushes its event log and latest snapshot to an external durable store (such as S3, R2, or a database) as a background operation after every N moves. This external backup means that if the actor host crashes or becomes unavailable, a new actor instance can be spun up on any available node and reconstruct full game state by loading the latest backup and replaying any events since the last flush. For platform-level outages, clients receive a clear "game temporarily unavailable" message and can retry; no game data is lost because the external backup provides the recovery point. This is a pragmatic level of resilience for a game system -- not mission-critical redundancy, but sufficient to ensure no player ever permanently loses a game in progress.

For connection resilience, the client persists the current sandbox state to local storage during a turn, so that if a WebSocket connection drops mid-manipulation, the player can reconnect and resume editing without losing work. The server handles reconnection by pushing the current authoritative game state to the newly connected client, and the client reconciles its local sandbox accordingly.

Authentication and authorization are enforced at the actor entry point: players authenticate via a standard JWT token (issued by a separate auth service with standard token rotation and short expiry times), and the actor verifies that the authenticated player ID matches the player whose turn it is before accepting a move. WebSocket subscriptions are scoped so that each player receives only the public board state and their own rack -- never another player's rack. Spectators receive board-only state updates. For games that attract many spectators, the actor can delegate fan-out to a lightweight pub-sub relay (e.g., a simple broadcast channel or managed service) rather than pushing to every WebSocket directly, keeping the actor's own workload bounded.

The event-sourced log provides built-in support for full game replay, spectator mode, undo/redo within a turn (client-side, since the sandbox is local), post-game analytics, and debugging. Events are versioned with a schema version field, and the replay logic includes lightweight migration transforms that upcast old event formats to the current schema during reconstruction.

### Advantages
- Near-zero infrastructure cost for idle games, since serverless actors only execute when moves are made, perfectly matching the bursty nature of turn-based play
- The declarative move diff protocol massively simplifies server validation -- no need to track or synchronize intermediate tile rearrangement states
- Platform-portable design: the game actor is a plain TypeScript class behind a thin adapter, deployable on Cloudflare, AWS, GCP, or self-hosted infrastructure with no rearchitecting
- Server-side AI functions written in pure TypeScript with a clean contract enable multiple difficulty levels and easy upgrades, with no native dependency or platform compatibility issues
- Event-sourced game log with periodic snapshots provides full replay, spectator mode, analytics, and debugging without unbounded replay latency
- External backup of event logs and snapshots ensures game state survives actor crashes or platform outages, providing practical fault tolerance without over-engineering
- Shared pure-function TypeScript validation module gives instant client-side feedback while keeping the server authoritative, with no runtime-specific coupling
- Explicit security model: JWT authentication with short-lived tokens, per-player state scoping, rate limiting, message size bounds, server-authoritative validation with tile ID verification, and no externally exposed AI endpoints
- Naturally supports asynchronous play (turns hours apart) since actors hibernate with near-zero cost between moves
- Spectator fan-out delegation keeps the game actor's workload bounded regardless of viewer count
