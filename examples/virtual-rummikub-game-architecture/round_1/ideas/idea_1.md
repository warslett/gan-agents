## Idea: CRDT-Based Optimistic Board Manipulation with WASM-Shared Game Logic

### Description

The most distinctive challenge in Rummikub compared to other board games is the free-form board manipulation phase: on each turn, a player can rearrange the entire table, splitting runs, recombining groups, and weaving in tiles from their rack, with the only constraint being that the board is valid when they hit "done." Most architectures treat this as either a purely client-side editing phase (with server validation at the end) or lock the board into a strict move-based protocol. This idea proposes a third path: use Conflict-free Replicated Data Types (CRDTs) to model the board as a collaboratively editable surface, combined with a single Rust/WASM game-logic core that runs identically on both client and server.

The architecture has three layers. First, a shared game-logic core written in Rust, compiled to WebAssembly. This single codebase handles tile validation, set recognition, initial-meld checking, scoring, and the ILP-based AI solver (using a lightweight solver like `good_lp` with the WASM-compatible HiGHS backend). Because the exact same binary runs on the server (via wasmtime or embedded in a Node/Deno host) and in the browser, there is zero risk of validation desync -- the perennial danger of event-sourcing architectures like boardgame.io where client and server might diverge. The WASM core exposes a clean FFI: `apply_move(state, action) -> Result<state, error>`, `validate_board(board) -> bool`, `solve_ai_turn(state, difficulty) -> action[]`, and `filter_state(state, player_id) -> visible_state`.

Second, the real-time layer uses a CRDT document (via Yjs or Automerge) to represent the board during the active player's manipulation phase. When a player starts their turn, the current board state is loaded into a CRDT document. The player drags tiles around freely -- every intermediate rearrangement is a CRDT operation that is synced in real time to all spectators and opponents via a lightweight WebSocket relay (no server logic needed, just forwarding). Other players see the active player thinking and rearranging in real time, creating a rich "watching over their shoulder" experience that mimics sitting at a physical table. Crucially, these intermediate states are NOT validated or authoritative -- they are purely visual. When the player clicks "done," the final CRDT document state is serialized and sent to the authoritative server, which runs `validate_board` through the WASM core. If invalid, the server rejects and the board snaps back to the start-of-turn snapshot (also stored in the CRDT history). If valid, the server commits the new state and advances the turn.

Third, the infrastructure layer is deliberately simple: a single Deno or Node.js server process hosts the WASM runtime, manages game rooms via a simple in-memory map, persists completed moves to a SQLite or PostgreSQL event log, and exposes a WebSocket endpoint. The CRDT relay can be a separate lightweight process or even a managed service (like Liveblocks or PartyKit) since it carries no game logic. For the frontend, a Svelte or SolidJS app consumes the CRDT document reactively -- since CRDTs naturally produce fine-grained change notifications, the UI updates are extremely efficient without needing a virtual DOM diffing layer. The tile board is rendered with a CSS Grid layout where each set is a flex row, and tiles are standard DOM elements styled with CSS (no Canvas needed for a tile game), using a library like dnd-kit or Pragmatic drag-and-drop for interaction.

The AI opponent is also elegant in this architecture. When it is the AI's turn, the server invokes `solve_ai_turn` in the WASM core, which uses the ILP solver to find the optimal tile placement and returns a sequence of actions. These actions are replayed into a CRDT document at a human-like pace (with configurable delays and even simulated "hesitation" for realism), so human players see the AI "thinking" and moving tiles around before committing, just like a real opponent. Difficulty levels are implemented by constraining the solver: easy mode uses a greedy heuristic, medium uses ILP but with a limited time budget, and hard uses full ILP optimization plus strategic tile-holding heuristics.

### Advantages
- Zero validation desync risk because the exact same WASM binary runs on client and server, eliminating the most common bug class in multiplayer game architectures
- Real-time board manipulation spectating via CRDTs gives a uniquely social, "physical table" feel that no other Rummikub implementation offers
- The CRDT relay carries no game logic, so it can be scaled independently and cheaply (or replaced with a managed service), while the authoritative server stays simple
- Rust/WASM game core enables embedding a performant ILP solver for AI directly in the shared logic, avoiding the need for a separate AI microservice
- AI opponents appear lifelike because their moves are streamed through the same CRDT visualization pipeline as human moves
- The architecture cleanly separates concerns: game rules (WASM core), real-time collaboration (CRDT layer), persistence (event log), and presentation (reactive frontend), making each independently testable and replaceable
- Svelte/SolidJS plus CRDT change notifications means the UI is highly performant without virtual DOM overhead, important for the complex drag-and-drop interactions Rummikub demands
- Event sourcing via the committed move log enables full game replays, undo history, and analytics without additional infrastructure
