## Idea: Serverless Event-Sourced Actor Architecture with Pluggable AI Microservices and Offline-First PWA Shell

### Description

Rather than running a persistent game server process, this architecture models each Rummikub game as a lightweight, event-sourced actor that lives in a serverless environment (such as AWS Lambda backed by DynamoDB, or Cloudflare Durable Objects). Each game actor maintains its state as an ordered log of moves (events), and its current state is reconstructed on demand by replaying those events. When a player submits a move, a cloud function is invoked that loads the game actor, validates the move against the reconstructed state, appends the event to the log, and fans out the updated state to connected clients via a managed WebSocket gateway (such as AWS API Gateway WebSocket API or Cloudflare WebSocket hibernation). This eliminates the need to keep server processes running between moves -- a natural fit for turn-based games where minutes may pass between actions -- and scales to thousands of concurrent games with near-zero idle cost.

The heart of the system is a declarative move protocol. During a player's turn, all tile manipulation -- dragging tiles between rack and board, splitting runs, forming new groups -- happens entirely on the client side within an isolated "sandbox" copy of the board state. The client does not stream incremental manipulations to the server. Instead, when the player commits their turn, the client computes a declarative move description: a structured diff specifying which tiles moved from rack to board, which tiles were rearranged between sets, and the final composition of every set on the board. This single atomic message is sent to the server actor, which independently validates it by checking that all sets are legal, no tiles have vanished, the initial meld requirement is met if applicable, and at least one new tile was played. If validation fails, the server rejects the move and the client reverts to the pre-turn snapshot. This approach radically simplifies the server: it only needs to understand "before state" and "proposed after state," not the messy intermediate steps of tile rearrangement.

AI opponents are implemented as independent microservices behind a simple API contract: given a game state (board sets, the AI player's rack, and known information), return a move in the same declarative diff format that human clients produce. This decoupling means AI implementations can be swapped, upgraded, or A/B tested without touching the game engine. A "beginner" AI service might use a fast greedy heuristic that only looks for direct placements. An "expert" AI service could use an ILP solver (via OR-Tools or GLPK compiled to a container) that finds the mathematically optimal tile placement each turn. A "strategic" AI could layer MCTS on top of the ILP solver to make longer-horizon decisions about which tiles to hold back. Because these are stateless HTTP services, they scale independently and can even be offered as a paid tier or community-contributed plugins. The game actor simply calls the appropriate AI endpoint when it is an AI player's turn, receives the move, validates it through the same pipeline as human moves, and appends it to the event log.

The client is built as a Progressive Web App with an offline-first design. The full game logic validation module (written in TypeScript) is shared between the serverless functions and the client bundle. This means players can start a local game against AI opponents that run entirely in the browser (using the greedy heuristic AI compiled to the client, or even a lightweight ILP solver via a WASM build of GLPK). When they want to play multiplayer, they sign in and the same UI seamlessly connects to the serverless backend. Game state can be serialized to IndexedDB for resume-later functionality, and the event-sourced log format makes it trivial to implement full game replay, undo during a turn, spectator mode (by subscribing to the event stream), and even asynchronous play where players take turns hours apart -- the serverless actor simply hibernates between moves and wakes on the next invocation.

### Advantages
- Near-zero infrastructure cost for idle games, since serverless functions only execute when moves are made, perfectly matching the bursty nature of turn-based play
- The declarative move diff protocol massively simplifies server validation -- no need to track or synchronize intermediate tile rearrangement states
- AI opponents are fully decoupled microservices with a clean contract, enabling multiple difficulty levels, independent scaling, easy upgrades, and even community-contributed AI implementations
- Event-sourced game log provides built-in support for full game replay, spectator mode, undo/redo, analytics, and debugging with zero additional engineering
- Offline-first PWA design lets players practice against AI without any server connection, then seamlessly transition to online multiplayer with the same interface
- Shared TypeScript validation logic between client and server eliminates the risk of validation divergence and enables instant client-side feedback before server round-trips
- Naturally supports asynchronous play (turns hours apart) since there is no persistent server process to keep alive between moves
- Horizontally scales to massive numbers of concurrent games without complex server orchestration, as each game is an independent actor with its own event stream
