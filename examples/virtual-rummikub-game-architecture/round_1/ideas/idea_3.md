## Idea: WebRTC Peer-to-Peer Mesh with Ephemeral Validation Relay and Client-Side Tiered AI

### Description

Instead of relying on a persistent game server process (whether monolithic, serverless, or actor-based), this architecture eliminates the always-on server entirely by building on a WebRTC peer-to-peer data channel mesh between players, combined with a stateless "validation relay" that is consulted only at turn boundaries. Each player's browser connects directly to every other player's browser via WebRTC reliable data channels. During a player's turn, their tile manipulations are streamed directly to all peers for real-time spectating at zero server cost. The only server-side component is a stateless HTTP endpoint (a simple cloud function or edge function) that receives a board snapshot at the end of each turn, validates it against the previous board state, and returns a signed validation receipt. This receipt is then gossiped to all peers as proof that the move was legitimate. The validation relay holds no state between requests -- the submitting client sends both the previous board snapshot (which all peers already hold and can verify the hash of) and the proposed new board state.

The peer-to-peer mesh is established through a lightweight signaling server (which can be a free-tier WebSocket relay like PeerJS Cloud or a minimal Cloudflare Worker). Once the WebRTC connections are established, the signaling server is no longer needed. Game state is held collaboratively: all peers maintain an identical copy of the full game state (minus other players' hidden racks), and state transitions are gated by the signed validation receipts from the relay. A simple hash-chain of board states ensures no peer can tamper with history -- each validation receipt includes a hash of the previous state, forming a verifiable chain. If any peer detects a hash mismatch, they can challenge and the game pauses for resolution.

For AI opponents, this architecture takes a radically different approach from server-side solvers. AI players run entirely in the browser using Web Workers, with a tiered difficulty system built on progressively more sophisticated algorithms sharing a common interface. Easy AI uses a greedy heuristic that finds direct placements without rearranging the board. Medium AI uses a bounded-depth search that considers simple rearrangements. Hard AI employs a full ILP solver compiled to WebAssembly (using a library like HiGHS-WASM or GLPK.js) running in a dedicated Web Worker with a configurable time budget. Because AI runs client-side, single-player and local-multiplayer games require absolutely zero server infrastructure -- not even the validation relay, since there is no adversarial context. This means a player can practice against AI opponents with zero network dependency, and the same game logic seamlessly transitions to peer-to-peer multiplayer when they want to play online.

The frontend is built as a single SPA using a reactive framework like Svelte or SolidJS (chosen for their lean runtime compared to React, which matters since this architecture pushes more computation to the client). The game logic core -- tile validation, set detection, initial meld checking, turn state machine -- is written as a pure, framework-agnostic TypeScript module that runs identically in the main thread (for UI validation feedback), in Web Workers (for AI), and in the validation relay cloud function. The drag-and-drop tile manipulation UI uses a canvas-based renderer (via a library like Konva or PixiJS) for smooth 60fps tile dragging even on large boards, with an HTML overlay for UI controls. Lobby and matchmaking are handled through a simple room-code system: one player generates a short code, shares it out-of-band, and other players join by entering it. The signaling server maps codes to peer connection offers.

### Advantages
- Near-zero ongoing server costs since game data flows directly between players via WebRTC, and the validation relay is a stateless function that only fires once per turn (likely costing fractions of a cent per game)
- AI games work entirely offline with no server dependency whatsoever, enabling a true play-anywhere experience including on flights or in areas with poor connectivity
- Real-time streaming of tile manipulations during a player's turn is naturally supported by the peer mesh without any server bandwidth, allowing spectators to watch rearrangements as they happen
- The hash-chain of signed validation receipts provides tamper-proof game history without needing a persistent database, and game replays can be reconstructed from the chain
- The tiered client-side AI system allows players to gradually increase difficulty, and the ILP-based hard AI running in WASM provides genuinely optimal play without requiring expensive server-side compute
- Horizontal scaling is effectively infinite since each game is a self-contained peer mesh with no central server bottleneck -- the architecture handles 10 games or 10,000 games identically
- The pure TypeScript game logic module shared across client, worker, and cloud function ensures validation consistency while being easily testable in isolation
- Graceful degradation: if WebRTC connections fail, the system can fall back to relaying messages through the signaling server as a WebSocket bridge, maintaining functionality at the cost of slightly higher latency
