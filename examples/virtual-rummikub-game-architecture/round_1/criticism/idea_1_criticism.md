## Evaluation: CRDT-Based Optimistic Board Manipulation with WASM-Shared Game Logic

### Scores

| Dimension | Score (1-10) | Justification |
|---|---|---|
| Feasibility | 4 | Combining CRDTs, Rust/WASM with ILP solvers, and cross-platform binary sharing introduces substantial integration risk and requires rare developer expertise. |
| Simplicity | 3 | Three distinct architectural layers, a CRDT relay, a WASM FFI boundary, an ILP solver, and event sourcing create a high number of interacting moving parts for a tile game. |
| Fault Tolerance | 4 | The proposal has almost no discussion of failure modes, relies on a single in-memory server process, and the CRDT layer introduces ambiguous recovery semantics. |
| Maintainability | 5 | The clean separation of concerns aids reasoning, but debugging across WASM FFI, CRDT sync, and WebSocket relay layers is genuinely difficult in practice. |
| Adaptability | 6 | The WASM core's clean FFI and separation of game logic from transport is a genuine strength, though the CRDT document schema would need reworking for significantly different game modes. |

### Critical Analysis

#### Feasibility (Score: 4)
The idea stacks multiple individually complex technologies -- Rust compiled to WASM, CRDTs (Yjs/Automerge), ILP solvers compiled to WASM (HiGHS), cross-runtime WASM hosting (wasmtime in Deno/Node), and reactive frontends -- into a single architecture. Each of these technologies has its own integration challenges, and combining them multiplies the risk. The claim that HiGHS runs in a WASM-compatible mode is not well established; ILP solvers are typically large C++ codebases with system-level dependencies, and getting them to compile cleanly to WASM with acceptable performance is a non-trivial research task, not an implementation detail. The idea assumes a developer team comfortable with Rust, WASM FFI, CRDT internals, and ILP modeling simultaneously -- this is an exceptionally rare skill set. The claim of "zero risk of validation desync" because the same binary runs on both sides ignores the practical reality that the CRDT document state must be serialized, transmitted, deserialized, and then fed into the WASM core, introducing multiple points where desync can occur due to serialization bugs, version mismatches, or timing issues. The idea also casually mentions "a library like dnd-kit or Pragmatic drag-and-drop" without addressing that complex free-form tile rearrangement with CRDT-synchronized state is a drag-and-drop UX challenge well beyond what these libraries handle out of the box.

#### Simplicity (Score: 3)
For what is fundamentally a turn-based tile game for 2-4 players, this architecture is remarkably complex. The system requires: a Rust codebase compiled to WASM, a WASM runtime on the server, a CRDT library on both client and server, a WebSocket relay (possibly a separate process or managed service), a separate authoritative game server, an event log in SQLite/PostgreSQL, and a reactive frontend framework. That is at minimum six distinct technology concerns. The CRDT layer is used solely to provide a "watching over their shoulder" spectating experience during one player's turn -- a feature that could be achieved far more simply by throttled WebSocket broadcasts of tile positions. The ILP solver for AI adds yet another complex dependency for a feature that could be approximated with much simpler heuristic algorithms. The idea introduces CRDTs -- a technology designed for multi-writer conflict resolution in distributed systems -- to solve a problem (one player moving tiles while others watch) that has exactly one writer and zero conflicts. This is a fundamental mismatch between the tool and the problem.

#### Fault Tolerance (Score: 4)
The proposal explicitly describes a "single Deno or Node.js server process" with "game rooms via a simple in-memory map." If this process crashes, all active game state is lost. There is no mention of how to recover from: server crashes mid-game, WebSocket disconnections during a player's manipulation phase, CRDT document corruption, failures in the WASM runtime, or the CRDT relay going down. The event log stores "completed moves," so incomplete turns are not recoverable. The CRDT layer introduces its own failure modes: what happens if a player's CRDT document diverges from the relay's state? What if the "snap back" on invalid board state fails because the CRDT history is corrupted or unavailable? The idea mentions using the CRDT history to roll back to start-of-turn state, but CRDT history compaction is a known source of data loss in Yjs/Automerge. There is no health checking, no graceful degradation, and no redundancy anywhere in the design.

#### Maintainability (Score: 5)
The clean FFI boundary of the WASM core (`apply_move`, `validate_board`, etc.) is a genuine strength for testability -- the game logic can be unit tested in isolation in Rust. However, when something goes wrong in production, a developer must trace through: JavaScript frontend code, CRDT document operations, WebSocket messages to a relay, WebSocket messages to the authoritative server, WASM FFI calls, Rust logic, and database writes. Debugging WASM is notoriously poor -- stack traces are often opaque, and stepping through Rust-compiled WASM in browser devtools is painful. CRDT debugging is similarly difficult because the internal state is an opaque operation log, not a human-readable data structure. The interaction between the CRDT layer and the authoritative WASM validation creates a particularly tricky debugging surface: if a board state is rejected as invalid after a player spends time manipulating it, determining whether the issue is in the CRDT serialization, the WASM validation logic, or the player's actual arrangement requires crossing multiple abstraction boundaries.

#### Adaptability (Score: 6)
The separation of game logic into a WASM core with a clean FFI is genuinely beneficial for adaptability. Adding new validation rules or scoring variants means modifying only the Rust code. However, the CRDT document schema is tightly coupled to the board representation -- introducing a fundamentally different game mode (e.g., team play with shared racks, or a board with different topology) would require redesigning the CRDT document structure, which ripples through the relay, the frontend rendering, and the serialization boundary with the WASM core. The ILP-based AI solver is also brittle to rule changes: any change to what constitutes a valid set or run requires reformulating the integer linear program, which is a specialized modeling task. The "difficulty levels" described (greedy, time-limited ILP, full ILP) are a reasonable concept but the claim that they are simple to implement assumes the ILP formulation itself is straightforward, which for Rummikub's combinatorial complexity is not the case.

#### Security (Score: 3)
The architecture has a glaring security gap: the CRDT relay is described as carrying "no game logic" and "just forwarding" -- but this means it is an unauthenticated message relay unless additional security is bolted on, which is not discussed. A malicious client could: flood the relay with garbage CRDT operations to disrupt other players' views, inject misleading tile positions to confuse opponents during their observation, perform denial-of-service against the relay, or attempt to manipulate the CRDT document state before it reaches the authoritative server. The proposal mentions no authentication mechanism for WebSocket connections, no authorization checks (can a player manipulate tiles when it is not their turn?), no rate limiting, and no input validation on CRDT operations. The `filter_state(state, player_id) -> visible_state` function in the WASM core suggests awareness of information hiding, but there is no discussion of how this is enforced -- if the full game state is loaded into a CRDT document on the client, a malicious client could inspect the CRDT internals to see hidden tiles. The idea does not address how rack privacy is maintained when the CRDT document presumably contains all tile positions.

#### Scalability (Score: 4)
The architecture explicitly uses "a single Deno or Node.js server process" with "game rooms via a simple in-memory map." This is a single point of bottleneck with no horizontal scaling path described. The claim that the CRDT relay "can be scaled independently and cheaply" is asserted without any mechanism: how are CRDT documents partitioned across relay instances? How do players in the same game room connect to the same relay? The mention of managed services like Liveblocks or PartyKit is hand-waving -- these services have their own limitations, pricing models, and latency characteristics that are not analyzed. Running an ILP solver per AI turn in the WASM runtime is CPU-intensive and would block the single-threaded Node/Deno event loop unless explicitly offloaded to a worker thread, which is not discussed. For a game that typically has 2-4 players per room, the per-game overhead of a CRDT document, a WASM runtime instance, WebSocket connections, and an ILP solver instance is substantial compared to simpler architectures. There is no estimate of how many concurrent games this architecture could support.
