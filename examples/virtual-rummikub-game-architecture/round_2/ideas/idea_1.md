## Idea: WASM-Shared Game Logic with Real-Time Board Spectating

### Description

The central challenge in virtual Rummikub is the free-form board manipulation phase: on each turn, a player can rearrange the entire table -- splitting runs, recombining groups, weaving in tiles from their rack -- with the only constraint being that the board is valid when they hit "done." Most architectures treat this as either a purely client-side editing phase (with server validation at the end) or lock the board into a strict move-based protocol. This idea proposes a focused two-layer architecture: a shared Rust/WASM game-logic core that runs identically on client and server, paired with a simple WebSocket-based real-time broadcast for board spectating.

**Layer 1: Shared WASM Game Core.** A single Rust codebase compiled to WebAssembly handles all game rules: tile validation, set/run recognition, initial-meld checking, scoring, and AI opponent logic. The same WASM binary runs on the server (via a worker thread in Deno/Node using wasmtime) and in the browser. This eliminates the most common bug class in multiplayer game architectures -- validation desync between client and server. The WASM core exposes a clean FFI: `apply_move(state, action) -> Result<state, error>`, `validate_board(board) -> bool`, `solve_ai_turn(state, difficulty) -> action[]`, and `filter_state(state, player_id) -> visible_state`. Client-side, the WASM core provides instant local feedback ("is this arrangement valid?") while the server remains the sole authority for committing state. To be clear about the desync claim: because the same compiled binary and the same serialized board representation are used on both sides, the only desync vector is a bug in the serialization layer itself, which is a single narrow surface that is straightforward to test exhaustively -- a far smaller risk than architectures where client and server run independently written validation logic.

**Layer 2: WebSocket Broadcast for Real-Time Spectating.** When the active player manipulates tiles, the client sends throttled snapshots of tile positions (at roughly 10 updates per second) over an authenticated WebSocket connection to the server. The server rebroadcasts these to other players in the room. This gives opponents a "watching over their shoulder" experience that mimics sitting at a physical table. Crucially, these intermediate states are purely visual and non-authoritative -- the server does not validate them. When the player clicks "done," the final board state is sent to the server, which runs `validate_board` through the WASM core. If invalid, the board snaps back to the start-of-turn snapshot (stored server-side). If valid, the server commits the new state and advances the turn. This approach achieves the same real-time social experience without the complexity of CRDTs, which are designed for multi-writer conflict resolution -- a problem that does not exist here since exactly one player writes at a time.

**Server Design and Fault Tolerance.** The server is a Deno or Node.js process that hosts the WASM runtime in a dedicated worker thread (preventing ILP solver CPU work from blocking the event loop), manages game rooms, and persists state to PostgreSQL. Game state is persisted after every committed turn, so server crashes only lose the in-progress manipulation phase of the current turn -- which can be cleanly recovered by resetting to the last committed board state and restarting the active player's turn. WebSocket disconnections during manipulation are handled identically: on reconnect, the client receives the last committed state and the player resumes their turn. Health checks and graceful shutdown are standard operational concerns addressed by running the server behind a process manager (e.g., PM2, systemd) with PostgreSQL as the durable state store.

**Horizontal Scaling.** Because game rooms are independent, horizontal scaling is straightforward: run multiple server instances behind a load balancer with sticky sessions (by room ID). Each instance handles a partition of active rooms. PostgreSQL serves as the shared state store, enabling room migration between instances if needed. For the broadcast relay, if spectating traffic becomes a concern, it can be offloaded to a lightweight pub/sub layer (e.g., Redis pub/sub or a managed WebSocket service) without affecting game logic.

**Security.** All WebSocket connections require authentication via a JWT token issued at game join time. The server enforces authorization on every message: only the active player can send board manipulation updates, and only for their current turn. Broadcast messages are filtered through `filter_state` so players never receive information about opponents' rack tiles. Rate limiting is applied to manipulation updates (capped at the 10/s throttle rate) and connection attempts. Input validation is performed on all incoming board states before they reach the WASM core -- malformed payloads are rejected at the WebSocket handler level. Since the broadcast only sends positions of face-up table tiles (not rack tiles), and the server filters all outgoing state, a malicious client cannot extract hidden information by inspecting network traffic.

**AI Opponents.** When it is the AI's turn, the server invokes `solve_ai_turn` in the WASM core on the worker thread. The AI uses a greedy heuristic-based solver for easy and medium difficulty (fast and simple to implement) and a backtracking search with pruning for hard difficulty. This avoids the dependency on an ILP solver compiled to WASM -- backtracking search is well-suited to Rummikub's constraint structure and is straightforward to implement in Rust without external solver dependencies. AI moves are replayed to spectating players as a sequence of tile-position updates at a human-like pace, so human players see the AI "thinking" and rearranging tiles before committing, just like a real opponent.

**Frontend.** A Svelte or SolidJS app renders the board using CSS Grid with flex rows for sets. Tiles are standard DOM elements styled with CSS (no Canvas needed for a tile game). Drag-and-drop uses a library like dnd-kit or Pragmatic drag-and-drop. The local WASM core provides instant validation feedback as the player rearranges tiles, highlighting invalid sets in real time. The throttled position broadcast is a simple JSON message of tile IDs and coordinates -- the receiving clients animate tile movements smoothly using CSS transitions.

### Advantages
- Zero meaningful validation desync risk because the exact same WASM binary runs on client and server, with a single narrow serialization surface that is easy to test exhaustively
- Real-time board manipulation spectating via throttled WebSocket broadcasts gives a social, "physical table" feel with minimal implementation complexity -- no CRDT overhead for a single-writer problem
- Clean two-layer separation (game rules in WASM core, transport via WebSocket) makes each layer independently testable and replaceable
- Rust/WASM game core enables a performant AI solver to run directly in the shared logic on a worker thread, avoiding both a separate AI microservice and event-loop blocking
- AI opponents appear lifelike because their moves are streamed through the same broadcast pipeline as human moves
- Server crash recovery is clean: PostgreSQL stores committed state after every turn, and in-progress manipulation is simply restarted since it carries no authoritative data
- Horizontal scaling via sticky-session load balancing over independent game rooms, with PostgreSQL as shared state and optional Redis pub/sub for broadcast offloading
- Authenticated and authorized WebSocket connections with rate limiting, input validation, and state filtering prevent cheating, information leakage, and abuse
- Heuristic and backtracking AI avoids the integration risk of compiling an ILP solver to WASM while remaining effective for Rummikub's constraint structure
