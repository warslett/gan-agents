## Idea: Serverless Event-Sourced Architecture with Declarative Move Protocol

### Description

This architecture models each Rummikub game as a lightweight, event-sourced actor running in a serverless environment (such as Cloudflare Durable Objects, which provide built-in WebSocket hibernation and co-located persistent storage). Each game actor maintains its state as an ordered log of moves (events). To avoid the well-known problem of replay latency growing with game length, the actor persists periodic snapshots -- for example, every 10 moves -- so that state reconstruction only replays events since the last snapshot rather than the entire history. When a player submits a move, the actor loads the most recent snapshot, replays only the handful of subsequent events, validates the move, appends the event to the log, and pushes the updated state to connected clients via the Durable Object's native WebSocket support. This eliminates the need for a separate managed WebSocket gateway and keeps the per-game infrastructure minimal. Serverless functions only execute when moves are made, matching the bursty nature of turn-based play where minutes may pass between actions.

The heart of the system is a declarative move protocol. During a player's turn, all tile manipulation -- dragging tiles between rack and board, splitting runs, forming new groups -- happens entirely on the client side within an isolated "sandbox" copy of the board state. When the player commits their turn, the client computes a declarative move description: a structured diff specifying which tiles moved from rack to board, which tiles were rearranged between sets, and the final composition of every set on the board. This single atomic message is sent to the server actor, which independently validates it by checking that all sets are legal, no tiles have appeared or vanished (every tile is accounted for by ID against the server's authoritative state), the initial meld requirement is met if applicable, and at least one new tile was played from the rack. If validation fails, the server rejects the move and the client reverts to the pre-turn snapshot. This approach radically simplifies the server: it only needs to understand "before state" and "proposed after state," not the messy intermediate steps of tile rearrangement. Crucially, the server treats the client as untrusted -- the declarative diff is the trust boundary, and the server performs full independent reconstruction and verification of the proposed board state using authoritative tile IDs, making it impossible for a malicious client to fabricate tiles, reference tiles outside their rack, or peek at other players' racks.

The game validation logic is written in TypeScript as a pure-function module with no runtime-specific dependencies, ensuring it runs identically on Cloudflare Workers and in the browser. The client uses this shared module to provide instant feedback during tile placement (highlighting invalid sets, showing meld point totals) before the server round-trip. However, the server's validation is always authoritative -- client-side validation is purely a UX convenience and never trusted.

AI opponents are implemented as difficulty-tiered functions behind a simple internal API contract: given a game state (board sets and the AI player's rack), return a move in the same declarative diff format that human clients produce. A "beginner" AI uses a fast greedy heuristic that only looks for direct placements. An "expert" AI uses a constraint solver (such as OR-Tools) to find optimal tile placement. These AI functions run server-side only, invoked directly by the game actor within the same serverless environment -- not as separate HTTP microservices. This eliminates network latency, removes an external failure point, and avoids exposing AI endpoints as an attack surface. If an AI computation exceeds a time budget, the game actor falls back to the greedy heuristic, ensuring turns always complete. Because the AI produces moves in the same declarative format as human players, adding new difficulty levels requires only deploying a new function variant -- the game engine and validation pipeline remain untouched.

For connection resilience, the client persists the current sandbox state to local storage during a turn, so that if a WebSocket connection drops mid-manipulation, the player can reconnect and resume editing without losing work. The Durable Object handles reconnection by pushing the current authoritative game state to the newly connected client, and the client reconciles its local sandbox accordingly. For asynchronous play (turns hours apart), the Durable Object hibernates between moves with near-zero cost and wakes on the next WebSocket connection or HTTP invocation.

Authentication and authorization are enforced at the Durable Object entry point: players authenticate via a standard JWT token, and the actor verifies that the authenticated player ID matches the player whose turn it is before accepting a move. WebSocket subscriptions are scoped so that each player receives only the public board state and their own rack -- never another player's rack. Spectators receive board-only state updates.

The event-sourced log provides built-in support for full game replay, spectator mode, undo/redo within a turn (client-side, since the sandbox is local), post-game analytics, and debugging. For event schema evolution as rules change over time, events are versioned with a schema version field, and the replay logic includes lightweight migration transforms that upcast old event formats to the current schema during reconstruction.

### Advantages
- Near-zero infrastructure cost for idle games, since serverless functions only execute when moves are made, perfectly matching the bursty nature of turn-based play
- The declarative move diff protocol massively simplifies server validation -- no need to track or synchronize intermediate tile rearrangement states
- Server-side AI functions with a clean contract enable multiple difficulty levels and easy upgrades, while eliminating the latency and failure modes of external HTTP calls
- Event-sourced game log with periodic snapshots provides full replay, spectator mode, analytics, and debugging without unbounded replay latency as games grow longer
- Shared pure-function TypeScript validation module gives instant client-side feedback while keeping the server authoritative, with no runtime-specific coupling
- Explicit security model: JWT authentication, per-player state scoping, server-authoritative validation with tile ID verification, and no externally exposed AI endpoints
- Naturally supports asynchronous play (turns hours apart) since Durable Objects hibernate with near-zero cost between moves
- Connection resilience through client-side sandbox persistence and server-driven reconnection state push, so players never lose in-progress tile arrangements
- Event versioning strategy ensures long-term maintainability as game rules and features evolve
